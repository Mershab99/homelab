# Hub Talos Bootstrap

Single-node Talos control plane on the Supermicro. This directory holds:

- `schematic.yaml` — Image Factory schematic; defines the system extensions baked into the ISO
- `machineconfig/patches/` — config patches applied to `talosctl gen config` output. The "network baked in" lives here.
- `secrets.yaml.sops` — SOPS-encrypted Talos PKI bundle. Generated by `task bootstrap:secrets`.

## Hardware assumptions

| NIC  | Role                                               |
|------|----------------------------------------------------|
| eno1 | Enslaved to `br-wan`. VyOS VM uses this for WAN.   |
| eno2 | Enslaved to `br-lan` (VLAN-aware). VyOS LAN trunk. |
| eno3 | Host management on ISP LAN (192.168.1.20/24).      |
| eno4 | Reserved spare.                                    |

The host's foot in VLAN 10 is `br-lan.10` at `10.10.10.20/24` (no default route — that's via eno3 → 192.168.1.1).

## First-boot procedure

```bash
# 1. Generate Talos secrets (one-time per cluster)
task bootstrap:secrets

# 2. Render Cilium bootstrap manifest
task bootstrap:render

# 3. Build hub ISO via Image Factory
task talos:hub:iso

# 4. Generate full machineconfig with patches applied
task talos:hub:config

# 5. Flash ISO to USB
sudo dd if=gen/iso/talos-hub.iso of=/dev/diskN bs=4M status=progress

# 6. Boot Supermicro from USB. It will land in maintenance mode with a DHCP IP.
#    Find that IP via the IPMI console.

# 7. Apply machineconfig (insecure on first boot)
task talos:hub:apply NODE=<dhcp-ip> INSECURE=true

# 8. Talos reboots. Find it again at 10.10.10.20 (its configured static IP) once
#    eno3 is up — verify with: talosctl --nodes 192.168.1.20 dashboard

# 9. Bootstrap etcd
talosctl --talosconfig gen/talosconfig \
  --nodes 192.168.1.20 --endpoints 192.168.1.20 \
  bootstrap

# 10. Wait for the node to go Ready (Cilium pods come up via inlineManifests)
talosctl --nodes 192.168.1.20 health

# 11. Get kubeconfig
talosctl --nodes 192.168.1.20 kubeconfig .kubeconfig

# 12. Bootstrap Flux against the repo
GITHUB_TOKEN=ghp_xxx GH_OWNER=mershab task flux:bootstrap
```

## Why network is "baked in"

Strictly speaking, the *ISO* carries only kernel + initramfs + extensions + kernel
cmdline. The network config (bridges, VLANs, IPs) lives in `machineconfig/patches/`
and is applied at first boot via `talosctl apply-config`.

We call this "baked in" because:

1. The patches are committed to the repo, so the network config is fully declarative.
2. `task talos:hub:config` produces a single `gen/talos-hub-machineconfig.yaml` that
   contains every interface, address, route, and bridge — first-boot apply gives a
   fully routable, Cilium-running node with no further intervention.
3. The Cilium bootstrap manifest is *literally* embedded into the machineconfig as
   `cluster.inlineManifests`, so the CNI comes up alongside kubelet without a
   separate `kubectl apply`.

## Re-applying after the cluster is up

```bash
task talos:hub:apply NODE=10.10.10.20  # secure mode, default
```

## Upgrading Talos

```bash
task talos:hub:upgrade NODE=10.10.10.20             # dry-run by default
task talos:hub:upgrade NODE=10.10.10.20 DRY_RUN=false  # actually run
```

`--preserve` is always passed so etcd state survives.
